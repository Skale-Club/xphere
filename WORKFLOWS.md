# Workflows — Authoring Guide

This document is the single source of truth for any agent (human, Copilot, or external coding agent like Claude Code) authoring a workflow in this codebase. If you only read one file before writing a workflow, read this one.

## One-page mental model

A **Workflow** has:

- **A trigger** — what starts the run. Types: `tool_call` (invokable by name from agents/webhooks), `event` (e.g. `meeting.confirmed`), `schedule` (cron), `manual`, `webhook_url`.
- **A graph** — one or more `nodes` connected by `edges`. Single-action workflows have exactly one action node.
- **A kind** — `tool` (single-action, invokable by name) or `flow` (multi-step DAG). Choose based on the trigger: `tool_call` → `tool`; everything else → `flow`.

That's the whole model. There is no separate "Automations" system — that name was retired in SEED-025; everything is a Workflow.

## File format

Workflows are declarative YAML. The same YAML format is used whether the workflow is written by a human in the canvas, generated by Copilot, authored by Claude Code, or shipped as a platform default in `supabase/seeds/workflows/`.

```yaml
name: SMS reminder 5min before meeting
description: When a meeting is confirmed, schedule an SMS to the attendee 5 minutes before start.
trigger:
  type: event
  event: meeting.confirmed
nodes:
  - id: wait
    kind: wait
    duration: "-5m"
    from: "{{meeting.starts_at}}"
  - id: notify
    kind: send_sms
    integration: twilio
    to: "{{meeting.attendee_contact.phone}}"
    body: "Your appointment is in 5 minutes: {{meeting.link}}"
edges:
  - from: trigger
    to: wait
  - from: wait
    to: notify
```

YAML round-trips losslessly to the JSON `workflow_versions.definition` field — no data loss either direction.

## Authoring rules (validator-enforced)

A workflow is "valid" if **all** of these hold:

1. **Trigger** — exactly one `trigger` block with a `type` that exists in the spec
2. **Nodes** — at least one node; every node has a unique `id` and a known `kind`
3. **Integrations** — every node that declares an `integration:` references an integration that is **connected** for the org (the spec is org-filtered, so disconnected integrations don't even appear)
4. **Variables** — every `{{ref}}` references a namespace that is in scope at that node. Scope grows as nodes execute: `trigger.*`/`input.*` are always available; trigger-specific namespaces (e.g. `meeting.*` for calendar triggers) are added by the trigger; previous nodes' outputs become available as `{{<node_id>.*}}` downstream
5. **Edges** — every edge references real node ids; no cycles; every node is reachable from `trigger`

Errors are **structured** — they include `path`, `code`, `message`, and `suggestion`. The `suggestion` field is engineered for LLM consumption so an agent can read it, fix the YAML, and re-submit.

## Validation

```bash
# CLI (local) — depends on SEED-026 Phase A
npm run workflows:validate path/to/workflow.yaml

# HTTP (dry-run against the org-filtered spec)
curl -X POST https://xphere.app/api/workflows/validate \
  -H "Content-Type: application/json" \
  -d '{"definition": <YAML-decoded JSON>}'
```

Validation **never** writes to the database. Use it freely during authoring.

## Submission

Two paths:

### Path 1 — runtime API (immediate publish)

```
POST /api/workflows
Content-Type: application/json

{ "definition": <YAML-decoded JSON> }
```

The endpoint validates first; if validation fails, the workflow is **not** created and the structured errors are returned. On success: a new `workflow` row with `kind='flow'` (or `kind='tool'` if the trigger is `tool_call`) plus an initial `workflow_versions` row.

### Path 2 — repo-tracked (ships with releases)

Commit the YAML to `supabase/seeds/workflows/`. Every PR runs `npm run workflows:validate-all` which fails the build if any seed file is invalid. On deploy, the seed loader inserts/upserts the workflows for every org.

Use Path 2 for **platform-default workflows** — booking confirmations, missed-call follow-up, etc. Use Path 1 for **per-tenant workflows** created at runtime.

## Where the spec lives

The authoritative list of triggers, nodes, integrations, and variables is **`/api/workflows/spec`** (auth-required, org-scoped). Three properties make it AI-friendly:

1. **Org-filtered** — disconnected integrations are not listed. An LLM literally cannot generate a workflow referencing an unavailable capability.
2. **Self-describing** — each trigger and node has a human-language `description`. Read these to know what the thing does.
3. **JSONSchema-typed** — each node's `params_schema` is JSONSchema, so a code agent can validate locally without round-tripping the API.

For one-off reads from a coding agent, the static portion of the spec (triggers + node catalog) is mirrored at `docs/workflows/spec.schema.json` in the repo. The **dynamic** portion (available_integrations per org) only exists at runtime.

## Common patterns

### A. Single-action workflow callable by an AI agent (`kind='tool'`)

```yaml
name: send_appointment_sms
description: Tool that any agent or webhook can call to send an SMS confirming an appointment.
trigger:
  type: tool_call
  config:
    tool_name: send_appointment_sms
nodes:
  - id: action
    kind: send_sms
    integration: twilio
    to: "{{input.to}}"
    body: "{{input.body}}"
edges:
  - from: trigger
    to: action
```

### B. Multi-step calendar reminder (`kind='flow'`)

See "File format" example above.

### C. Branching workflow

```yaml
trigger:
  type: event
  event: meeting.no_show
nodes:
  - id: was_paid
    kind: condition
    expression: "{{meeting.was_paid}} == true"
  - id: refund_sms
    kind: send_sms
    integration: twilio
    to: "{{meeting.attendee_contact.phone}}"
    body: "We missed you — we'll refund within 48h."
  - id: rebook_sms
    kind: send_sms
    integration: twilio
    to: "{{meeting.attendee_contact.phone}}"
    body: "We missed you — book again: {{meeting.event_type.slug}}"
edges:
  - { from: trigger,   to: was_paid }
  - { from: was_paid,  to: refund_sms, when: "true" }
  - { from: was_paid,  to: rebook_sms, when: "false" }
```

## Things that will **not** work

These are common failure modes — the validator catches all of them with `suggestion` text:

- **Referencing a disconnected integration** — won't appear in the spec for the org; node will fail validation with `integration_unavailable`
- **Variables from the wrong scope** — `{{contact.phone}}` is not available on a `schedule` trigger; use the spec's per-trigger `variables` list
- **Cycles** — Workflow A trigger → action → back to A. Use `wait` + linear flow instead
- **Floating nodes** — every node must be reachable from `trigger`
- **Duplicate node ids** — must be unique within a workflow
- **Unknown node kinds** — only what's in the spec for this org

## For Claude Code / external coding agents

When asked to author a workflow:

1. Read this file
2. Read or fetch the spec — locally `docs/workflows/spec.schema.json`, or via `GET /api/workflows/spec` if running against a live org
3. Look at `.planning/workflows/examples/` for a similar pattern
4. Write the YAML
5. Run `npm run workflows:validate <file>` — if errors, fix them based on each `suggestion` field and re-validate; iterate until clean
6. Commit to `supabase/seeds/workflows/` (for platform-default) or POST to `/api/workflows` (for runtime)

The validator is the contract. If it passes, the workflow is publishable. There are no implicit conventions beyond what the validator checks.
