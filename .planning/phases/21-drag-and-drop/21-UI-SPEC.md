- Background: `bg-primary/10` (10% opacity of `--primary`)
- Ring: `ring-1 ring-inset ring-primary/40` (40% opacity of `--primary`, inset)

- Replaces the existing `bg-muted/30` rest state while active; reverts immediately on `dragOverFolderId` reset

**Folder drag-origin visual (folder being dragged):**
- `opacity-50` on the `SortableFolderHeader` row when `isDragging === true` (from `useSortable`)

- No `DragOverlay` rendered for folder drags

**Tool drag-origin visual (tool row being dragged):**
- `opacity-40` + `bg-muted/20` on the `DraggableToolRow` when `isDragging === true` (from `useDraggable`)
- Faded in place to indicate the tool is being moved, not copied

**DragOverlay chip (tool drag only):**
- Background: `bg-background` (white) with `border` and `shadow-md` — floats above the table surface
- Opacity: `opacity-90` — slightly transparent to show content behind cursor

**Grip icon color:**
- At rest (before row hover): `opacity-0` (invisible)
- On row hover (`group-hover/row:opacity-100`): `text-muted-foreground`
- On grab active (`active:cursor-grabbing`): `text-foreground`
- Transition: `transition-opacity`

## Interaction Contracts

### IC-01: Folder Reorder via Drag

- **Draggable element:** The `GripVertical` handle in `SortableFolderHeader` (existing handle — already wired to `useSortable` listeners).
- **What changes in Phase 21:** Add `data: { type: 'folder' }` to the `useSortable({ id, data: { type: 'folder' } })` call so `handleDragEnd` can branch correctly.
- **Visual during drag:**
  - Origin row: `opacity-50` (applied via `isDragging` from `useSortable`).
  - No `DragOverlay` rendered for folder drags — the opacity ghost is the only visual signal.
  - Other folder rows shift position using the existing `arrayMove` optimistic update (already implemented).
- **Drop behavior:** On `handleDragEnd`, when `active.data.current?.type === 'folder'`:
  1. `arrayMove(orderedFolders, oldIndex, newIndex)` — optimistic state update.
  2. `reorderFolders(reordered.map(f => f.id))` — new server action, batch-updates `position` for all top-level folders.
  3. `toast.error('Failed to save folder order.')` on failure (no rollback in Phase 21 — `router.refresh()` on next navigation corrects stale state).
- **Scope:** Top-level folders only. Subfolder drag reorder is out of scope.
- **Collision detection:** `closestCenter` (unchanged).

### IC-02: Tool Drag — Initiation

- **Draggable element:** `GripVertical` icon (`h-3 w-3`) absolutely positioned in the first `TableCell` of each tool row.
- **Visibility:**
  - Default: `opacity-0`
  - On row hover (`group-hover/row:opacity-100`): visible as `text-muted-foreground`
  - Cursor: `cursor-grab`, `active:cursor-grabbing`
  - `aria-label="Drag to move to folder"`
- **Implementation pattern:** Overlay approach — `position: absolute; left: 4px; top: 50%; transform: translateY(-50%)`. The first `TableCell` receives `className="relative group/row"`. No new `<TableCell>` column is added to avoid colSpan changes.
- **Hook:** `useDraggable({ id: tool.id, data: { type: 'tool' } })` in new `DraggableToolRow` component.
- **On drag start:** `onDragStart` handler sets `activeDragType = 'tool'` and `activeDragTool = tool` (full tool object from `toolConfigs`).
- **Tool row during drag:** `opacity-40 bg-muted/20` on the origin `TableRow`.

### IC-03: Folder Header Drop Target — Highlight

- **Trigger:** A tool is being dragged (`activeDragType === 'tool'`) AND the cursor is near a folder or subfolder header.
- **How highlight is tracked:** `onDragOver` handler reads `over?.id` and sets `dragOverFolderId` state. No secondary `useDroppable` hook — `useSortable` already registers folder headers as droppables; `closestCenter` will resolve `over.id` to the nearest folder header id.
- **Visual applied to `SortableFolderHeader` and `SubfolderHeader`:**
  - `isDropTarget` prop: `boolean` — equals `dragOverFolderId === folder.id` (or `sub.id`).
  - Highlighted className: `bg-primary/10 ring-1 ring-inset ring-primary/40` (replaces `bg-muted/30`).
  - At-rest className: `bg-muted/30 hover:bg-muted/40` (unchanged from Phase 20).
  - Transition: none (highlight is instant; debouncing would create lag in a DnD interaction).
- **Reset:** `dragOverFolderId` is set to `null` in both `handleDragEnd` and `handleDragCancel`.

### IC-04: Tool Drop onto Folder Header

- **Drop behavior:** On `handleDragEnd`, when `active.data.current?.type === 'tool'` and `over` is not `null`:
  1. Guard: if `tool.folder_id === over.id`, reset state and return (no-op — already in this folder).
  2. Call `moveToolToFolder(toolId, over.id as string)` — new focused server action.
  3. On success: `toast.success('Tool moved.')` + `router.refresh()` (required because `toolConfigs` is prop-derived from the server; local state cannot be updated without a re-fetch).
  4. On failure: `toast.error(result.error)`.
- **Drop behavior when `over` is `null`** (dropped in empty space): reset state, no action, no toast.
- **Result:** Tool disappears from its previous position and appears in the new folder immediately after `router.refresh()` completes. No optimistic local move (tool state is more complex than folder order — `router.refresh()` is the correct approach).

### IC-05: DragOverlay — Tool Name Chip

- **Rendered when:** `activeDragType === 'tool'` and `activeDragTool !== null`.
- **Content:** `activeDragTool.tool_name` — the tool's name string.
- **Appearance:**
  ```
  div.bg-background.border.rounded.px-3.py-1.5.text-sm.font-mono.shadow-md.opacity-90
  ```
- **Position:** Follows cursor automatically via `DragOverlay` (no manual positioning needed).
- **For folder drags:** `DragOverlay` renders `null` — the `opacity-50` ghost on the folder row is sufficient.

### IC-06: Drag Cancel

- **Trigger:** User presses `Escape` during a drag (keyboard cancel via @dnd-kit's `KeyboardSensor`).
- **Handler:** `onDragCancel` on `DndContext` — calls `resetDragState()` which clears `activeId`, `activeDragType`, `activeDragTool`, `dragOverFolderId`.
- **Visual:** All highlights and opacity changes revert immediately.
- **No toast** on cancel — no action was taken.

### IC-07: State Additions to ToolsTable

New state variables added in Phase 21 (in addition to all Phase 20 state):

| State Variable | Type | Default | Purpose |
|---------------|------|---------|---------|
| `activeId` | `string \| null` | `null` | ID of the item currently being dragged |
| `activeDragType` | `'folder' \| 'tool' \| null` | `null` | Distinguishes drag type for `handleDragEnd` branching |
| `activeDragTool` | `ToolConfigWithIntegration \| null` | `null` | Full tool object for `DragOverlay` chip rendering |
| `dragOverFolderId` | `string \| null` | `null` | Folder/subfolder id currently under a tool drag; drives `isDropTarget` prop |

## DraggableToolRow — Component Specification

`DraggableToolRow` is a new named component defined outside `ToolsTable` (to avoid re-creation on each render, matching the `SortableFolderHeader` and `SubfolderHeader` pattern from Phase 20).

**Props:**
```typescript
interface DraggableToolRowProps {
  tool: ToolConfigWithIntegration
  row: Row<ToolConfigWithIntegration>
}
```

**Structure:**
```
TableRow [ref=setNodeRef, className=cn("group/row", isDragging && "opacity-40 bg-muted/20")]
  For each visible cell (i = 0..N-1):
    TableCell [className=i===0 ? "relative group/row" : undefined]
      IF i === 0:
        span [absolute grip handle — see IC-02]
      flexRender(cell.column.columnDef.cell, cell.getContext())
```

**colSpan unchanged:** The grip is overlaid inside the existing first cell, not added as a new `<TableCell>`. `colSpan` on folder header rows remains `columns.length` — no change.

## Server Action Contract

Two new server actions required. Executor MUST implement these before wiring the UI.

| Action | File | Signature | Purpose |
|--------|------|-----------|---------|
| `reorderFolders` | `src/app/(dashboard)/tools/actions.ts` | `(orderedIds: string[]) => Promise<void \| { error?: string }>` | Batch-updates `position` for each top-level folder by its index in `orderedIds`; calls `revalidatePath('/tools')` |
| `moveToolToFolder` | `src/app/(dashboard)/tools/actions.ts` | `(toolId: string, folderId: string \| null) => Promise<void \| { error?: string }>` | Updates `tool_configs.folder_id` for a single tool; calls `revalidatePath('/tools')` |

**`reorderFolders` guard:** Only update folders where `parent_id IS NULL` (server-side safety — prevents subfolder positions being corrupted if an invalid id is passed).

**`moveToolToFolder` null support:** Accepts `null` for `folderId` (moves tool to Ungrouped). Not exposed via drag in Phase 21 (the Ungrouped section is not a droppable), but the action supports it for future use.
