<objective>
Add conditional config fields to the tool config form for the send_sms and custom_webhook action types, and fix the server actions so that custom_webhook saves do not write an empty string to the NOT NULL integration_id column.

Purpose: Admins currently cannot configure send_sms (no Twilio integration picker hint) or custom_webhook (no URL/method/headers/body fields) from the UI — they would need direct database access. This plan closes that gap. It also patches a data-integrity bug where the custom_webhook save path passes an empty integrationId to the DB insert/update, which Postgres rejects on the NOT NULL UUID column.

Output: An updated tool-config-form.tsx that conditionally renders action-specific fields, extends the zod schema with an optional config sub-object, and passes config values through to the existing server actions. An updated actions.ts where integration_id is only included in DB inserts/updates when integrationId is a non-empty string.
</objective>

**Part A — Fix actions.ts (do this first, it is a blocker)**

In `src/app/(dashboard)/tools/actions.ts`, fix both `createToolConfig` and `updateToolConfig` so that `integration_id` is only written to the DB when `data.integrationId` is a non-empty string. The `tool_configs.integration_id` column is `NOT NULL UUID` — passing an empty string causes a Postgres rejection.

In `createToolConfig`, replace the insert object with a spread that conditionally includes `integration_id`:

```typescript
const { error } = await supabase.from('tool_configs').insert({
  organization_id: member.organization_id,
  tool_name: data.toolName,
  action_type: data.actionType as Database['public']['Enums']['action_type'],
  ...(data.integrationId && data.integrationId.length > 0
    ? { integration_id: data.integrationId }
    : {}),
  fallback_message: data.fallbackMessage,
  config: (data.config ?? {}) as Json,
  folder_id: data.folder_id ?? null,
  labels: data.labels ?? [],
})
```

In `updateToolConfig`, replace the update object with the same conditional spread:

```typescript
const { error } = await supabase
```

Replace the current `toolConfigSchema` with:

```typescript
const toolConfigSchema = z.object({
  toolName: z
    .string()
    .min(1, 'Tool name is required')
    .max(100, 'Tool name must be 100 characters or fewer'),
  actionType: z.enum([
    'create_contact',
    'get_availability',
    'create_appointment',
    'send_sms',
    'knowledge_base',
    'custom_webhook',
    'manychat_set_field',
    'manychat_add_tag',
    'manychat_trigger_flow',
    'manychat_send_message',
    'google_contacts_create',
    'google_contacts_update',
    'google_contacts_find',
    'google_contacts_delete',
  ]),
  integrationId: z.string().optional().nullable(),
  fallbackMessage: z
    .string()
    .min(1, 'Fallback message is required')
    .max(500, 'Fallback message must be 500 characters or fewer'),
  folder_id: z.string().optional().nullable(),
  labels: z.array(z.string()),
  config: z
    .object({
      url: z.string().optional(),
      method: z.string().optional(),
      headers: z.string().optional(),
      body: z.string().optional(),
    })
    .optional(),
}).superRefine((data, ctx) => {
  // integrationId required for action types that need an integration
  const requiresIntegration = data.actionType !== 'custom_webhook'
  if (requiresIntegration && !data.integrationId) {
    ctx.addIssue({
      code: z.ZodIssueCode.custom,
      message: 'Please select an integration',
      path: ['integrationId'],
    })
  }
  // url required when actionType is custom_webhook
  if (data.actionType === 'custom_webhook' && !data.config?.url?.trim()) {
    ctx.addIssue({
      code: z.ZodIssueCode.custom,
      message: 'Webhook URL is required',
      path: ['config', 'url'],
    })
  }
})
```
