# Operator v1.4 Requirements

## Milestone: v1.4 — Chat System Refactor

**Goal:** Improve maintainability and UX of the chat system: split oversized modules, clarify data boundaries, add realtime to admin inbox, fix broken tests, add conversation search.

---

## v1.4 Requirements

### REFACTOR — Module Decomposition

- [x] **REFACTOR-01**: `src/lib/chat/stream.ts` is split into focused modules — SSE encoder, provider routing (OpenAI/Anthropic/OpenRouter), RAG/knowledge lookup, action engine dispatch — each under 200 lines with a single concern; existing widget streaming behavior is unchanged
- [x] **REFACTOR-02**: `src/components/chat/chat-area.tsx` is split into `ChatHeader`, `MessageList`, `MessageBanner`, `MessageComposer` components — each under 150 lines; existing ChatArea continues to render identically with no visual regression
- [x] **REFACTOR-03**: All existing tests, the production build, and the live chat widget continue to work after the refactor — no behavioral changes

### BOUNDARY — Chat Data Boundary

- [x] **BOUNDARY-01**: A documentation file (`.planning/codebase/chat-data-boundary.md`) explains when the `chat_sessions` table is written vs when `conversations` is written, who owns each record, and how widget messages flow into the admin inbox
- [x] **BOUNDARY-02**: Code comments in the relevant files (`src/lib/chat/session.ts`, `src/lib/chat/persist.ts`, `src/app/api/chat/[token]/route.ts`) reference the boundary doc so future readers find it from the source

### REALTIME — Admin Inbox Real-Time Updates

- [x] **REALTIME-01**: When a new conversation arrives (widget chat, Instagram DM, or Messenger), it appears in the admin inbox `ConversationList` within 5 seconds without a manual refresh, using Supabase Realtime postgres_changes subscription
- [x] **REALTIME-02**: When a new message arrives in an open conversation, it appears in the `ChatArea` message list in real-time without polling
- [x] **REALTIME-03**: Realtime subscription is scoped per-org via RLS so admins only receive events for their own organization's conversations
- [x] **REALTIME-04**: Subscription cleanup happens correctly on unmount — no zombie connections after navigation

### TESTFIX — Broken Tests

- [x] **TESTFIX-01**: `tests/chat-persist.test.ts` — both currently-failing tests are fixed by updating to the current schema/table names; the full file passes (`npx vitest run tests/chat-persist.test.ts` exits 0 with no failures)
- [x] **TESTFIX-02**: `tests/action-engine.test.ts` ACTN-02 test (currently failing) is investigated and either fixed or formally retired with a comment explaining why; the full vitest suite has 0 unrelated failures

### SEARCH — Conversation Search

- [x] **SEARCH-01**: Admin can type free text in a search box above the `ConversationList`; conversations whose customer name or `last_message` match (case-insensitive substring) are shown; non-matching are hidden
- [x] **SEARCH-02**: Search composes with the existing channel filter and bot-state filter — applying all three narrows results
- [x] **SEARCH-03**: Search query is debounced (300 ms) to avoid re-rendering on every keystroke
- [x] **SEARCH-04**: Empty search clears the filter and shows all conversations again

---

## Future Requirements (Deferred)

- Stream.ts: pluggable LLM provider registry (would let new providers be added without core changes)
- ChatArea: virtualized message list for very long conversations
- Server-side full-text search using Postgres `tsvector` for searching message bodies (current SEARCH-01 is client-side, only over loaded conversations)
- Saved replies / message templates in the inbox
- Internal admin notes per conversation
- Bulk actions on conversations (mark resolved, archive, delete)

## Out of Scope

- New chat surfaces or features — this is purely a refactor + UX polish milestone
- Changes to the embeddable widget script (`src/widget/` and `src/reviews-widget/`) — bundles untouched
- Changes to the Vapi voice flow — separate concern
- Schema migrations — refactor must work with the current schema as-is
