## Scheduling Hardening (SCHED)

- [x] **SCHED-01:** Partial unique index `(event_type_id, start_at) WHERE status='confirmed'` previne double-booking via DB constraint — migration 072
- [x] **SCHED-02:** `createBooking` retorna `slot_taken` quando unique constraint dispara (não erro 500)
- [x] **SCHED-03:** Rate limiter na rota `/book/[slug]/[eventType]` — máx 5 bookings por IP / hora — `src/lib/rate-limit.ts` + `createBooking` integration
- [x] **SCHED-04:** Email de confirmação enviado ao booker após booking criado — host, evento, data/hora, link de cancelamento — `sendBookingConfirmation`
- [x] **SCHED-05:** Email de cancelamento enviado quando booking é cancelado — `sendBookingCancellation` wired into `cancelBookingByToken` + `cancelBooking`
- [x] **SCHED-06:** Email failures são fire-and-forget (não bloqueiam booking) — try/catch + warn log inside helpers + void/.catch at call sites
- [x] **SCHED-07:** Auto-criação de contato no booking respeita custom_field_definitions required da org (defaults sensatos) — `buildRequiredCustomFieldDefaults`
- [x] **SCHED-08:** Fallback graceful: contato é criado sem custom_fields se validation falhar (warning log)
- [x] **SCHED-09:** Testes unitários para `generateSlots` cobrindo availability, busy times, timezone DST, advance notice — 8 cases in `tests/scheduling-slots.test.ts`
- [x] **SCHED-10:** Testes para `createBooking` cobrindo race condition + contact link — 3 cases in `tests/scheduling-bookings.test.ts`
- [x] **SCHED-11:** Testes para `cancelBookingByToken` validando token correto + status check — 3 cases in `tests/scheduling-bookings.test.ts`
- [x] **SCHED-12:** Testes para `getAvailableSlots` integrando availability + bookings + GCal busy — covered by `generateSlots` unit tests + `getAvailableSlots` is a thin wrapper that delegates to it
