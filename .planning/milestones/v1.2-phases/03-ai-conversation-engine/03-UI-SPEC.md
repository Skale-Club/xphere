# Phase 3 — UI Design Contract

> Visual and interaction contract for the AI Conversation Engine.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope Note

Phase 3 is a **pure backend phase**. It replaces the stub response at step 7 of
`src/app/api/chat/[token]/route.ts` with a real streaming AI engine. No new
dashboard pages, admin views, or widget UI are added here.

The UI contract this phase owns is the **SSE stream interface** — the exact
event shape the Phase 4 widget JavaScript must parse, and the observable
interaction states (streaming, tool-calling, error, degraded) that Phase 4
must render. All dashboard-level design tokens are carried forward unchanged
from the established shadcn/Tailwind v4 system.

## Stream Interface Contract

This is the primary UI output of Phase 3. Phase 4's widget JavaScript must
implement a parser for this exact protocol (source: CONTEXT.md decisions D-01
through D-04).

### Response Headers

| Header | Value |
|--------|-------|
| Content-Type | `text/event-stream` |
| Cache-Control | `no-cache` |

### SSE Event Shape

All events are newline-delimited JSON objects. Each line is a complete JSON
value followed by `\n`. The widget reads the stream line-by-line and parses
each line as JSON.

#### Event: `session`
Emitted once, as the first line in every stream, before any tokens.

```json
{"event":"session","sessionId":"<uuid>"}
```

Widget behavior: Store `sessionId` for subsequent requests in this conversation.

#### Event: `token`
Emitted once per streamed text chunk from the LLM.

```json
{"event":"token","text":"<chunk>"}
```

Widget behavior: Append `text` to the in-progress assistant message bubble.
Do not add whitespace between chunks — the LLM delivers natural spacing.

#### Event: `tool_call`
Optional. Emitted when the model decides to invoke an org tool mid-stream.
The stream pauses tokens while `executeAction()` runs.

```json
{"event":"tool_call","name":"<action_type>"}
```

Widget behavior: Show a transient "thinking" indicator or `<action_type>` label
(e.g. "Looking up availability..."). Clear the indicator when the next `token`
event arrives.

#### Event: `done`
Emitted once, as the final line in every stream.

```json
{"event":"done"}
```

Widget behavior: Mark the assistant message as complete. Re-enable the send
button. Stop any loading indicators.

### Graceful Degradation Event Sequence

When the org has no API keys configured (decision D-12), the stream emits
exactly this sequence and no HTTP error is returned:

```
{"event":"session","sessionId":"<uuid>"}
{"event":"token","text":"This assistant is not yet configured. Please contact the site owner."}
{"event":"done"}
```

Widget behavior: Render the degradation message exactly like a normal assistant
reply — no special error styling. The message itself communicates the issue.

### Observable Interaction States

These are the states Phase 4's widget must support based on this phase's stream:

| State | Trigger | Widget Expectation |
|-------|---------|-------------------|
| Sending | User taps Send | Disable send button, show spinner in send button |
| Streaming | First `token` event received | Show assistant bubble with cursor, tokens appending live |
| Tool calling | `tool_call` event received | Show secondary indicator: "Checking [name]..." below the in-progress bubble |
| Complete | `done` event received | Remove cursor, remove indicator, re-enable send button |
| HTTP error (401) | Invalid token | Show static error: "Chat is unavailable. Token invalid." — no retry |
| HTTP error (400) | Malformed request | Show static error: "Message could not be sent. Please try again." |
| Network error | Fetch throws | Show static error: "Connection lost. Please check your connection." |

Note: 500 errors are returned as `{"error":"Internal server error"}` JSON from
the existing catch block. Widget must distinguish `text/event-stream` from
`application/json` by checking `Content-Type` on the response.
