<objective>
Wire the two existing billing failure catch blocks (Stripe webhook processing, Copilot credit debit RPC) to write a row into the existing `event_logs` table via the existing `log()` helper, so failures are queryable/alertable instead of silently swallowed to `console.error`. No new schema, no new admin UI, no new helper — this is purely additive wiring at two call sites, plus test coverage proving both call sites fire correctly and a documented manual check that the existing `/admin/logs` viewer surfaces both.

Purpose: Closes the observability gap in the billing-robustness milestone — when Stripe webhook processing throws, or `meterDebit()` fails open, an admin currently has no way to know without querying the database directly.
Output: Two production call sites emit structured `event_logs` rows on failure; two existing test files gain mocked assertions proving it; a manual verification note confirms the existing `/admin/logs` page surfaces both without any UI change.
</objective>

```typescript
export interface LogEntry {
  event_type: string          // Dot-namespaced, e.g. 'webhook.failed', 'credit_debit.failed'
  source: string               // Origin subsystem tag
  severity?: LogSeverity        // 'debug'|'info'|'warn'|'error'|'fatal' — default 'info'
  status?: LogStatus            // 'ok'|'failed'|'retried'|'skipped' — default 'ok'
  org_id?: string               // omit for platform-level events
  correlation_id?: string
  actor_type?: string           // 'system'|'user'|'agent'|'webhook'
  actor_id?: string
  payload?: Record<string, unknown>
  error_message?: string
  duration_ms?: number
}
export async function log(entry: LogEntry): Promise<void>  // NEVER throws — call with `void log({...})`, do not await, do not wrap in try/catch
```
