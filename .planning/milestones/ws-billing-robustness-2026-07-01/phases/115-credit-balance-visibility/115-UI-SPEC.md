## Color

Reuses the project's existing CSS variable tokens (`src/app/globals.css`) — no new colors introduced.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `var(--bg-primary)` | TopBar background, popover content background (via `--popover` → `--bg-secondary`) |
| Secondary (30%) | `var(--bg-secondary)` / `var(--bg-tertiary)` | Popover surface, mobile Quick Actions tile background |
| Accent (10%) | `var(--accent)` (#6366F1) | Reserved for: the credit icon (`Sparkles`) in its normal/healthy state, and the "Manage billing" link text/hover |
| Warning (low balance) | `amber-500` (Tailwind utility, matching the existing convention in `plan-usage-card.tsx`'s `over` state) | Badge/icon color when balance is low but not zero |
| Destructive (zero balance) | `var(--destructive)` → `var(--danger)` (#DC2626 light / #EF4444 dark) | Badge/icon color when balance is zero |

Accent reserved for: the `Sparkles` trigger icon in its default (non-low, non-zero) state, and the "Manage billing" link inside the popover. Never used for body text or backgrounds elsewhere in this component.

**Low/zero visual state (CRB-04):** the indicator has exactly 3 states, threshold-driven by `totalUsd` relative to `includedAllowanceUsd`:
- **Healthy** (`totalUsd > 0.2 * includedAllowanceUsd`, or no allowance concept applies): icon + text in default `text-text-secondary` / `--accent` icon tint, no badge.
- **Low** (`0 < totalUsd <= 0.2 * includedAllowanceUsd` — 20% threshold, matching the "near limit" spirit of `plan-usage-card.tsx`): icon and balance number render in `amber-500`, small dot badge (no number) top-right of the trigger icon, same badge geometry as `NotificationBell`'s unread badge (`absolute -right-0.5 -top-0.5`, `h-4 min-w-4`, `ring-2 ring-bg-primary`) but amber-filled with no label text (a plain dot, not a count).
- **Zero** (`totalUsd <= 0`): icon and balance number render in `var(--destructive)`, same dot badge in destructive color.

This 20% threshold is Claude's discretion (not specified in CONTEXT.md) — chosen for consistency with the existing `plan-usage-card.tsx` "over/near-limit" pattern in the same billing surface. The executor may adjust the exact percentage during implementation if a plan-level threshold is later formalized, but must keep exactly 3 visual states (healthy / low / zero) — no more.

## Component Contract (phase-specific, beyond the template's generic categories)

Since this phase is a single, tightly-scoped component addition, the following implementation-facing details are part of the contract (sourced from CONTEXT.md `## Decisions`, which already answered nearly every open question):

**Component name:** `CreditsIndicator` (new file, e.g. `src/components/billing/credits-indicator.tsx`), following the same `'use client'` + Popover + Realtime-channel pattern as `NotificationBell`.

**Desktop placement:** `TopBar`'s desktop actions row, between `NotificationBell` and `ThemeToggle` (CONTEXT.md specifies "between ThemeToggle and OrgSwitcher" — read literally that positions it after ThemeToggle; either adjacent slot in the existing `gap-1.5` flex row is acceptable as long as it sits in the icon-button cluster, not inside `OrgSwitcher`).

**Trigger button:**
- `Button variant="ghost" size="icon-sm"` (exact `NotificationBell` trigger pattern)
- Icon: `Sparkles` from lucide-react (already used by `CreditsCard`), sized `h-[15px] w-[15px]` (matches `NotificationBell`'s `Bell` icon sizing exactly)
- Badge overlay only appears in Low/Zero states (see Color section) — Healthy state shows no badge, matching `NotificationBell`'s "no badge when unreadCount is 0" behavior

**Popover:**
- `PopoverContent align="end" sideOffset={8} className="w-80 p-0"` — exact match to `NotificationBell`
- Internal layout: header row (icon + "Copilot credits" label) → balance number + breakdown line (reusing `CreditsCard`'s exact typographic treatment, scaled to `text-xl` instead of `text-2xl` for popover density) → conditional low/zero inline note → "Manage billing" link (styled as a text link in `--accent`, not a full `Button`, to keep the popover compact) to `/settings/billing`
- Clicking "Manage billing" closes the popover (`setOpen(false)`) before navigation, consistent with `NotificationBell`'s `onClose` pattern in `NotificationList`

**Data flow:**
- `getCopilotBalance(activeOrgId)` called in `(dashboard)/layout.tsx`, alongside (not replacing) the existing `entitlements` resolution
- A plan-includes-credits boolean is also resolved server-side (`PLAN_CATALOG[planKey]?.copilotIncludedUsd > 0`, OR an existing non-empty `copilot_credit_balances` row) and passed down — this boolean, not `isBillingEnforced()`, controls whether `CreditsIndicator` renders at all (CRB-03)
- Both values passed as new `TopBar` props, threaded to `CreditsIndicator` in both desktop and `MobileMenu` render paths

**Realtime:**
- `supabase.channel(\`copilot-credits:${orgId}:${instanceId}\`)` (mirrors `NotificationBell`'s `notifications:${userId}:${instanceId}` naming), `.on('postgres_changes', { event: 'UPDATE', schema: 'public', table: 'copilot_credit_balances', filter: \`org_id=eq.${orgId}\` }, handler)`, `.subscribe()`, cleanup via `supabase.removeChannel(channel)` on unmount — exact structural match to `NotificationBell`'s effect
- Requires new migration enabling Realtime publication on `copilot_credit_balances` (`ALTER PUBLICATION supabase_realtime ADD TABLE public.copilot_credit_balances`), following migration 024's precedent — this is a backend task tracked in the plan, not a UI-SPEC concern, but noted here since it's required for CRB-02 to function
- No pulse/highlight animation on update — the number updates instantly in place (explicit CONTEXT.md decision, "Claude's discretion, kept simple")

**Mobile:**
- Added as a 4th tile in `MobileMenu`'s "Quick actions" `grid-cols-3` section — CONTEXT.md says "matching that 3-icon grid pattern" alongside Dial pad / Notifications / Theme. Since this adds a 4th item to a 3-column grid, the executor should either: (a) keep `grid-cols-3` and let the 4th tile wrap to a new row (simplest, matches existing grid behavior with `hasPhoneNumber` conditionally present/absent), or (b) this is Claude's/executor's discretion since CONTEXT.md doesn't specify wrap behavior explicitly — default to (a), no new grid column count.
- Same invisible-overlay-trigers-popover technique as the existing Notifications tile (`<div className="absolute inset-0 opacity-0"><CreditsIndicator .../></div>` over a visible icon+label tile) — reuse this exact technique rather than inventing a new mobile interaction.
- Tile hidden entirely (not rendered) when the plan-includes-credits boolean is false, consistent with desktop hiding behavior (CRB-03).
