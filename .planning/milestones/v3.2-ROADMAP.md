# Milestone v3.2: Credits Visibility & Metering Architecture

**Status:** ✅ SHIPPED 2026-07-01
**Phases:** 114-117
**Total Plans:** 7
**Workstream:** `billing-robustness` (ran parallel to v3.1 Websites Lead Ingestion)

## Overview

Stripe billing, subscriptions, entitlements, and the Copilot credit wallet (included + topup buckets, ledger, atomic RPCs) already shipped and were production-ready before this milestone. This milestone hardened that foundation rather than rebuilding it: it made credit balance visible to users, generalized the credit-debit path into a reusable interface any future feature can plug into, backfilled automated test coverage across the billing surface, and closed the observability gap so webhook/debit failures are no longer silent. No new feature was wired to actually debit credits — Copilot remains the only debiting feature by design (deferred to v2 requirements MET-05..08).

**Requirements**: MET-01, MET-02, MET-03, MET-04

## Milestone Summary

**Key Decisions:**

- Architecture-first scope: build the reusable metering interface and observability wiring now; defer wiring workflows/campaigns/calls to actually consume credits to a future milestone (v2 requirements MET-05..08)
- Ran as a parallel GSD workstream (`billing-robustness`) rather than blocking on v3.1 (Websites Lead Ingestion), which had already completed its 3 phases but wasn't yet formally closed
- Reused existing platform primitives instead of building new surfaces: `event_logs` + `/admin/logs` for observability (Phase 117), `NotificationBell`'s Popover/Realtime pattern as the direct template for `CreditsIndicator` (Phase 115)
- Migrations 1224-1226 applied directly via the Supabase Management API rather than `npx supabase db push`, because the local Supabase CLI had no authenticated session for the project — schema changes verified live via direct queries each time

**Issues Resolved:**

- Fixed a `server-only` import leaking into a client bundle (extracted `credits-visibility.ts` as a client-safe module) — Phase 115
- Fixed RPC error-message extraction so Supabase error objects surface a real message instead of `"[object Object]"` — Phase 117
- Recovered cleanly from a host machine restart that interrupted Phase 114's executor mid-plan (work was already committed; only the final SUMMARY/state bookkeeping needed reconstruction)

**Issues Deferred:**

- Supabase CLI auth/migration-history desync (remote history recorded migrations 1224-1226 under synthetic timestamp versions since they were applied via the Management API) — needs `supabase login` + `supabase migration repair` as a separate operational task
- 5 Phase 115 UI behaviors (indicator render, popover, mobile tile, live update, low/zero states) approved on code/build/test evidence rather than a live browser click-through, since the connected Chrome extension had no network route to the local dev server — tracked in the archived `115-HUMAN-UAT.md`

**Technical Debt Incurred:**

- No component-render test infrastructure exists in this repo (`vitest.config.ts` environment: `'node'`, no `@testing-library/react`) — pre-existing repo-wide gap, not introduced by this milestone, but means CRB-01/CRB-02's UI rendering has no automated coverage
- ~64 pre-existing, unrelated test failures across 37 files elsewhere in the repo (auth, accounts, agents, etc.) — confirmed via git stash/git log to predate this milestone, logged in each phase's `deferred-items.md`, out of scope
