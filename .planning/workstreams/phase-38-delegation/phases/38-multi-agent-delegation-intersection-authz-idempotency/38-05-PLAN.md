# Plan 38-05: SSE Partner Events + Widget Badge UI

## Objective

The streaming path already emits `partner_start` and `partner_done` events (added in Plan 38-03
via the `emit` closure passed to `buildPartnerTools`). This plan adds:
1. A check of `organizations.delegation_visibility` in the streaming path to conditionally
   suppress partner events for orgs that prefer a cleaner UI
2. Widget badge display in `playground-chat.tsx` for `partner_start` / `partner_done` events

## Tasks

### Task 38-05-01: Gate partner SSE events on delegation_visibility in run-agent.ts

<read_first>

<action>
In `runAgentStreaming` inside `src/lib/agent-runtime/run-agent.ts`, after resolving the agent
and before building partner tools, fetch the org's `delegation_visibility` setting:

```typescript
// DELEG-08: Check delegation_visibility for this org
const delegVisClient = createServiceRoleClient()
const { data: orgRow } = await delegVisClient
  .from('organizations')
  .select('delegation_visibility')
  .eq('id', orgId)
  .single()
const delegationVisible = (orgRow?.delegation_visibility ?? 'visible') === 'visible'
```

Then pass a conditional emit to `buildPartnerTools`:
```typescript
// Only pass emit if delegation events should be visible (DELEG-08)
emit: delegationVisible ? emit : undefined,
```

This means the `emit` parameter in `buildPartnerTools` is only set when `delegation_visibility = 'visible'`.
The partner tool execute still calls `runAgentBlocking` regardless â€” only the SSE events are gated.

Note: `delegationVisible` should be fetched only once per streaming invocation. Place this
fetch after the agent is resolved and the invocation row is inserted (so it doesn't block the
early guard checks).

For `runAgentBlocking` (non-streaming path): `partner_start`/`partner_done` events are SSE-only
and do not apply â€” no change needed.
</action>

### Task 38-05-02: Widget badge display in playground-chat.tsx (DELEG-08)

<action>
In `src/components/chat/playground-chat.tsx`, extend the SSE event parsing loop to handle
`partner_start` and `partner_done` events by inserting transient internal badge messages.

**1. Add a `partnerBadge` field to the `Message` interface:**
```typescript
interface Message {
  id: string
  role: 'user' | 'assistant'
  content: string
  streaming?: boolean
  isPartnerBadge?: boolean  // true for partner delegation status badges
}
```

**2. In the SSE parsing loop**, after handling `token`, add:
```typescript
} else if (event.event === 'partner_start' && typeof event.partnerName === 'string') {
  const badgeId = crypto.randomUUID()
  const label = `Asking ${event.partnerName}...`
  setMessages((prev) => [
    ...prev,
    {
      id: badgeId,
      role: 'assistant',
      content: label,
      isPartnerBadge: true,
    },
  ])
} else if (event.event === 'partner_done' && typeof event.partnerName === 'string') {
  // Update the matching partner_start badge to show completion
  setMessages((prev) =>
    prev.map((m) =>
      m.isPartnerBadge && m.content === `Asking ${event.partnerName}...`
        ? { ...m, content: `${event.partnerName} responded` }
        : m
    )
  )
}
```

**3. In the JSX rendering**, add a visual distinction for partner badge messages.
In the `sendMessage` function's render loop, find where messages are rendered and add special
treatment for `isPartnerBadge: true` messages. The `playground-chat.tsx` component renders its
own message list inline (not using `MessageList` component). Add a conditional branch:

```typescript
// In the messages.map() render inside the JSX return:
if (message.isPartnerBadge) {
  return (
    <div key={message.id} className="flex justify-center my-2">
      <span className="inline-flex items-center gap-1.5 px-3 py-1 rounded-full text-xs font-medium bg-violet-50 text-violet-700 border border-violet-200 dark:bg-violet-950/30 dark:text-violet-300 dark:border-violet-800/50">
        <svg className="w-3 h-3 animate-pulse" fill="currentColor" viewBox="0 0 8 8">
          <circle cx="4" cy="4" r="3" />
        </svg>
        {message.content}
      </span>
    </div>
  )
}
```

This must be inserted before the existing `message.role === 'user'` / `'assistant'` branches.
</action>
