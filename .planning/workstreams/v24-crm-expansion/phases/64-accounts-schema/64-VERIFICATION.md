### Deviation Verification

**Plan 64-01 Rule-1 deviation:** Added `DELETE FROM public.opportunities WHERE contact_id IS NULL AND account_id IS NULL;` before adding the CHECK constraint (lines 133-135 of the migration).

- ✓ **Documented:** 64-01-SUMMARY.md § "Deviations from Plan > Auto-fixed Issues > 1. [Rule 1 - Bug]" describes the orphan opportunity (`f3f72a90-df6b-4061-8f19-d0e3cedfb990`, "teste", value 0, contact_id NULL, from 2026-05-17), the failure mode (SQLSTATE 23514 on first `db push`), and the fix.
- ✓ **Does NOT compromise SC3:** SC3 says "Attempting to insert an opportunity with both `contact_id IS NULL` and `account_id IS NULL` is rejected by the DB CHECK constraint." Test 2b in `tests/accounts-schema.test.ts:127-186` actively attempts an orphan INSERT under a freshly-seeded org and expects Postgres to reject with the constraint name — passed (523ms). The DELETE only addressed legacy data that pre-dated the constraint; the CHECK is what now blocks new orphans.
- ✓ **Idempotent:** Per the migration's own comment at lines 130-131, "re-running deletes zero rows because no future write can produce a row satisfying both NULLs once the CHECK is in place."
- ✓ **Plan 64-01-SUMMARY commit hash matches:** `9790e2d` is the single commit for Plan 64-01 and contains both the original schema AND the deviation fix (per the SUMMARY note, "the migration file as committed already contains the reordered cleanup logic").
