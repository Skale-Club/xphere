For jsonb columns, the three columns in `custom_field_definitions` (`default_value`, `options`, `validation`) are all nullable, so they map to `unknown | null` (the Phase 69 zod schema will narrow these at call sites). This deliberately differs from the `accounts.custom_fields` typing (`Record<string, unknown>`) because that column has a NOT NULL default of `'{}'::jsonb` and is always an object, while these three jsonb columns can be a primitive (default_value), an array (options for select), or null.

Per the `accounts` Update convention (omit immutable fields), the Update block here omits: `id`, `org_id`, `entity`, `created_by`, `created_at`. The `entity` column is omitted from Update because changing a definition's entity is semantically nonsensical (a definition belongs to one entity for life â€” to "change" it, archive and recreate). All other columns are optional.
