<design_constraints>
From CONTEXT.md — bake these in, do not deviate:
- Core functions are PLAIN async functions (NO `'use server'` directive). Server Actions must be top-level exports of `'use server'` modules and cannot be closures from a factory; therefore the shared logic lives here as plain functions and each module keeps a thin wrapper LATER.
- Every function takes the item/table context explicitly: signature shape is `fn(ctx, ...args)` where `ctx = { supabase, entityType, itemTable }` (a single context object keeps call sites clean and lets wrappers bind once).
- Folder queries always add `.eq('entity_type', ctx.entityType)` so one org's workflow folders never collide with its email_template folders.
- Item operations write `folder_id` / `position` on `ctx.itemTable` (column names are consistent across entities by decision).
- Return the same `ActionResult<T>` shape used by the existing actions: `{ ok: true; data: T } | { ok: false; error: string }`.
- Do NOT call `revalidatePath` in core (that is Next-server-action concern; wrappers do it). Do NOT call `getUser()` here (auth gating is the wrapper's job) — the core receives an already-authenticated `supabase` client.
- Do NOT repoint any existing consumer. Nothing imports the core this phase.
</design_constraints>

    ```typescript
    // Phase 114 (UFE-02): universal foldering core.
    //
    // Plain async functions (NOT 'use server') implementing the full folder + item
    // organization contract against the shared `public.folders` table, parameterized
    // by entity_type + item table. Per-module 'use server' wrappers (phases 115-117)
    // bind a FolderingContext and call these; the core carries the ONE copy of the
    // CRUD/cascade/move/reorder logic. No revalidatePath, no auth here — the wrapper
    // supplies an already-authenticated supabase client and handles Next concerns.

    import type { createClient } from '@/lib/supabase/server'
    import type { Database } from '@/types/database'

    export type FolderEntityType = 'workflow' | 'project' | 'tool' | 'email_template'
    export type SupabaseServerClient = Awaited<ReturnType<typeof createClient>>
    export type FolderRow = Database['public']['Tables']['folders']['Row']

    export interface FolderingContext {
      supabase: SupabaseServerClient
      entityType: FolderEntityType
      /** Item table whose rows carry folder_id + position, e.g. 'workflows'. */
      itemTable: string
    }

    export type ActionResult<T = void> =
      | { ok: true; data: T }
      | { ok: false; error: string }
    ```

    Implement these functions (port each from the referenced source, adding `.eq('entity_type', ctx.entityType)` to every folder query and using `ctx.itemTable` for item writes). Use `ctx.supabase.from('folders')` for folder ops; use `(ctx.supabase.from(ctx.itemTable) as any)` where the dynamic table name defeats the generated types (keep the `as any` narrow — only on the dynamic-table query builder, not on results).

    1. `export async function listFolders(ctx: FolderingContext): Promise<ActionResult<FolderRow[]>>`
       - `from('folders').select('*').eq('entity_type', ctx.entityType).order('position').order('created_at')`. Return rows.

    2. `export async function createFolder(ctx: FolderingContext, input: { name: string; color?: string | null; icon?: string | null; parent_id?: string | null }): Promise<ActionResult<FolderRow>>`
       - Trim name; `name_required` if empty. Resolve org via `ctx.supabase.rpc('get_current_org_id')`; `no_active_org` if null. Compute next sibling position filtered by `entity_type` + `parent_id`. Insert with `entity_type: ctx.entityType`, `created_by` (fetch via `ctx.supabase.auth.getUser()` OR accept an optional `createdBy` — prefer: fetch the user id here since ctx has the client). On `error.code === '23505'` return the friendly duplicate message. Return inserted row.

    3. `export async function renameFolder(ctx: FolderingContext, id: string, input: { name: string }): Promise<ActionResult<void>>`
       - Trim; `name_required` if empty. `update({ name }).eq('id', id).eq('entity_type', ctx.entityType)`. Map 23505 to friendly message.

    4. `export async function updateFolderMeta(ctx: FolderingContext, id: string, input: { color?: string | null; icon?: string | null }): Promise<ActionResult<void>>`
       - Build patch from defined color/icon; no-op ok if empty. `update(patch).eq('id', id).eq('entity_type', ctx.entityType)`.

    5. `export async function reorderFolders(ctx: FolderingContext, orderedIds: string[]): Promise<ActionResult<void>>`
       - For each id set `position: index` (Promise.all of updates, each `.eq('id', id).eq('entity_type', ctx.entityType)`). Friendly failure message.

    6. `export async function moveFolder(ctx: FolderingContext, id: string, parent_id: string | null): Promise<ActionResult<void>>`
       - Reject `parent_id === id` (`cannot_nest_in_self`). `update({ parent_id }).eq('id', id).eq('entity_type', ctx.entityType)`.

    7. `async function collectDescendantFolderIds(ctx: FolderingContext, rootId: string): Promise<string[]>` (private helper)
       - Port the bounded (16-level) breadth-first descendant walk from folders.ts, adding `.eq('entity_type', ctx.entityType)` to the child query.

    8. `export async function archiveFolder(ctx: FolderingContext, id: string): Promise<ActionResult<void>>`
       - Collect descendant folder ids. `update({ archived_at: now, is_active: false }).in('folder_id', folderIds).is('deleted_at', null).is('archived_at', null)` on `ctx.itemTable`. (Item lifecycle columns are assumed to exist on the item table when a module actually uses archive — this phase only defines the logic; it is not invoked here.)

    9. `export async function deleteFolder(ctx: FolderingContext, id: string, opts: { cascadeChildren?: boolean } = {}): Promise<ActionResult<void>>`
       - Check for child folders (filtered by entity_type); if present and `!cascadeChildren` return `folder_has_children`. Collect descendants, soft-delete items (`update({ deleted_at: now }).in('folder_id', folderIds).is('deleted_at', null)` on `ctx.itemTable`), then hard-delete the folder row (`.eq('id', id).eq('entity_type', ctx.entityType)`; ON DELETE CASCADE handles nested folders).

    10. `export async function moveItemToFolder(ctx: FolderingContext, itemId: string, folderId: string | null): Promise<ActionResult<void>>`
        - Port moveWorkflowToFolder: compute next position at destination on `ctx.itemTable` (`.eq('folder_id', folderId)`), then `update({ folder_id: folderId, position: nextPosition }).eq('id', itemId)`.

    11. `export async function reorderItemsInFolder(ctx: FolderingContext, _folderId: string | null, orderedIds: string[]): Promise<ActionResult<void>>`
        - Port reorderWorkflowsInFolder: Promise.all of `update({ position: index }).eq('id', id)` on `ctx.itemTable`. Friendly failure message.

    Keep every folder query scoped by `.eq('entity_type', ctx.entityType)`. Do NOT import `revalidatePath`. Do NOT add `'use server'`. Ensure the file type-checks (the `as any` should be confined to the dynamic-item-table `.from(ctx.itemTable)` builder).
