core so a consumer whose item table names its folder-linkage column something other
than `folder_id` (Projects uses `space_id` on `projects`) can delegate to the core

```typescript
export interface FolderingContext {
  supabase: SupabaseServerClient
  entityType: FolderEntityType
  /** Item table whose rows carry the folder-linkage column + position, e.g. 'workflows'. */
  itemTable: string
  /**
   * Name of the folder-linkage column on `itemTable`. Defaults to 'folder_id'.
   * Projects override this with 'space_id' (projects.space_id). Workflows, Tools,
   * and Email keep the default.
   */
  itemFolderColumn?: string
}
```

2. In `moveItemToFolder`, resolve the column name once and use it for BOTH the
   tail-position read (`.eq(...)`) and the update payload. Replace the function body
   so the hardcoded `'folder_id'` string literal and the `{ folder_id: folderId, ... }`
   key both come from `ctx.itemFolderColumn ?? 'folder_id'`:

```typescript
export async function moveItemToFolder(
  ctx: FolderingContext,
  itemId: string,
  folderId: string | null,
): Promise<ActionResult<void>> {
  const col = ctx.itemFolderColumn ?? 'folder_id'

  // Append at the end of the destination by default.
  const { data: tail } = await itemTable(ctx)
    .select('position')
    .eq(col, folderId as unknown as string)
    .order('position', { ascending: false })
    .limit(1)

  const nextPosition = ((tail?.[0]?.position as number | undefined) ?? -1) + 1

  const { error } = await itemTable(ctx)
    .update({ [col]: folderId, position: nextPosition })
    .eq('id', itemId)

  if (error) return { ok: false, error: error.message }
  return { ok: true, data: undefined }
}
```

Do NOT change `reorderItemsInFolder` (it never touches the folder column — only
`position` and `id`). Do NOT touch `archiveFolder` / `deleteFolder` in this plan;
their `.in('folder_id', folderIds)` calls stay hardcoded 'folder_id' because Projects
does its archive/delete cascade in its own wrapper (spaces.ts) against `projects.space_id`
directly, not via the core's archive/delete (see plan 116-03). Keeping those untouched
avoids over-generalizing and keeps this change minimal.
