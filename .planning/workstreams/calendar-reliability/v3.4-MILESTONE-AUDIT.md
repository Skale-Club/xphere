## 2. Cross-Phase Integration - E2E flows traced in code

**a. Public booking to validation to creation to lifecycle to Google id/Meet link.**
createBooking to resolveAndValidateSlot (P126: active event type + availability grid +
organizer conflict + Google busy) to insert status=BOOKING_STATUSES[0] to
createCalendarEvent persists google_event_id (P129) to emitCalendarEvent(meeting.scheduled)
(P127) to matched workflows via findMatchingWorkflows. COHERENT.

**b. Xkedule inbound to status guard to lifecycle to single event.**
webhook KNOWN_XKEDULE_STATUSES guard (P129) before DB access to mapStatus (completed to
showed) to existing row: mutable update (status excluded) then runXkeduleTransition to
canonical (P127); new row: insert + calendarEventFor emit. Update-error returns before any
emit (LIFE-03). COHERENT.

**c. Reminder: booking to tick scheduler to workflow dispatch.**
calendar-tick reads calendar_tick_watermark, resolveWindow transition-safety guard
(self-seeds on missing row), offset-shifted booking scan, per-booking
computeStartsInTargetMinute dedup insert to emitCalendarEvent consuming P127 event/scope
contract; watermark advances only on zero-released pass. COHERENT.

**d. Cancel: dashboard / MCP / public token to canonical cancelBooking to one meeting.cancelled.**
All three (cancelBooking bookings.ts:252, MCP bookings.ts:216, cancelBookingByToken
bookings.ts:972) call transitionCancelBooking with allowedFrom of [confirmed]; idempotent
no-op on already-cancelled (no second event). COHERENT.

## 5. Consolidated Tech-Debt / Follow-Up Inventory

### Deploy-time operator actions (MUST run at merge - from 128-06 + phase human_verification)
1. Watermark re-seed: immediately after Coolify deploys, run
   UPDATE public.calendar_tick_watermark SET scanned_to = now(); so the first tick on the new
   offset-derived keys does not catch-up-dispatch the migration-apply to code-deploy gap.
2. Confirm CRON_SECRET is set in Coolify AND the GitHub Actions repo secret. The route now
   503s when unset - a missing secret takes the reminder cron fully offline. Verify a plain
   GET of https://xphere.app/api/cron/calendar-tick returns 401 (not 503); a Bearer-authed
   call returns 200 with starts_in_scan/ended_scan fields.

### Real functional wart (recommend fixing or explicitly accepting)
3. google_meet bookings create TWO Google Calendar events + divergent stored id. In
   createBooking/createBookingInternal, createCalendarEvent (plain event, P129) and
   createMeetingLink (Meet-enabled event, P130) both POST to calendars/primary/events,
   producing two events on the host calendar for one booking. Worse, the two paths write
   google_event_id in OPPOSITE order: createBooking ends with the Meet event id (Meet block
   runs last), createBookingInternal ends with the plain event id (createCalendarEvent runs
   last, overwriting the Meet id). Neither requirement fails (SYNC-01 stores an id; SYNC-04
   puts the Meet link on meeting_url), and this path is exactly the still-deferred HUMAN-UAT
   #6, so it has never been exercised end-to-end. Top item for the pending google_meet UAT.

### Deferred bug consciously carried from Phase 126 (documented in booking-validation.ts header)
4. Display-path multi-window crash: getAvailableSlots + getDebugSlots (bookings.ts:352-357,
   443-448) still use .maybeSingle() on user_availability and pass a single window into
   generateSlots - they throw on any day with 2+ configured availability windows. The write
   path (resolveAndValidateSlot) is fixed (queries as an array); the booker-facing slot
   display is not. RESEARCH.md Pitfall 5.

### Minor coherence/doc drift (low priority)
5. transition.ts confirmBooking dead-branch writes the Meet id into location_data (line 296)
   with a different shape than the dedicated google_event_id column. Unreachable in
   production (bookings are born confirmed, so confirmBooking always short-circuits at the
   not-transitioned guard) - documented as intentional, but inconsistent with the live path.
6. Xkedule reschedule of an existing mirror row updates start_at/end_at via the raw mutable
   update with no meeting.rescheduled emission unless the status also changed. Pre-existing;
   arguably outside SYNC-02 (status semantics). Minor.
7. Stale comments / docs: calendar-tick.yml and route.ts still describe the window as now to
   now+1min; it is now watermark-bounded. ROADMAP.md truncates at Phase 130 (missing its
   Success Criteria / Plans subsections) though the phase completed 6/6.

### Pre-existing unrelated test red (track, not caused by this milestone)
8. tests/action-engine.test.ts - 8 failures in the /api/vapi/tools webhook route

   diff (127 deferred-items.md). Maintenance pass on the Vapi tools route + test recommended.
9. tests/workflows/*.test.ts - 2 pre-existing TS errors outside the app tsconfig (npm run
