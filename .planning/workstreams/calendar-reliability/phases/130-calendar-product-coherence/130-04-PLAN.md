<objective>
Close the SYNC-04 root blocker identified in 130-RESEARCH.md: `event-type-form.tsx` (the only admin UI for event types) has no field at all for `allowed_location_kinds` (the `text[]` column migration 089 added) — it only sets the legacy 3-value `location_type` enum. Because `createEventType`'s insert never explicitly sets `allowed_location_kinds`, it always takes the DB default `ARRAY['video']`. This single gap is why none of the 9 SEED-028 structured location kinds are reachable from the dashboard today, independent of each kind's individual backend readiness.

Per-kind downstream wiring is already correct for 5 kinds (`client_address`, `custom_address`, `phone_call`, `custom_phone`, `custom_link` — `location-resolver.ts`, `booking-form.tsx`, and `emails.ts` all already handle them) plus `google_meet` (completed in a companion plan for this phase, 130-05). `zoom`/`whereby` have zero backend integration (excluded — see 130-RESEARCH.md's per-kind verdict table) and `store_location` needs a larger lift than this phase's effort budget allows (excluded per the locked orchestrator decision for this phase — data preserved, `default_store_location_id` stays dormant).

Purpose: let admins actually configure the meeting-location kinds that already work end-to-end, without exposing the ones that don't.
Output: a new "Allowed meeting locations" multi-select on the event type form, validated server-side against exactly the 6 reachable kinds.
</objective>
