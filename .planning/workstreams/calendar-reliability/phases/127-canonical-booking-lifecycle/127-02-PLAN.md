<objective>
Fix `src/lib/calendar/scope.ts::buildMeetingScope` — the function every calendar workflow's `{{meeting.*}}` variables come from — which has selected a column that does not exist (`event_types.name`; the real column is `title`) since it was written, silently returning `{ data: null }` for the event type on every single call ever made in production. This makes `meeting.title`, `meeting.event_type.name`, and `meeting.event_type.slug` always wrong (`meeting.title` always renders the literal fallback `"Meeting"`) on every calendar-triggered workflow today. It also finishes the permanently-stubbed `meeting.organizer` object (`{ user_id: null, name: null, email: null }` today, unconditionally), populating it from the event type's host user when resolvable.

Purpose: LIFE-04 requires calendar workflow payloads to expose documented meeting variables consistently — a silently-broken column select and a permanent stub are the opposite of that. This plan does not change the MeetingScope interface's field names (external contract for workflow template authors is unchanged) — only the values populating it.
Output: `src/lib/calendar/scope.ts` selects the real column and populates `organizer`; `tests/calendar-scope.test.ts` (new) covers both fixes plus regression coverage for the rest of `buildMeetingScope`'s existing behavior.
</objective>

```typescript
async function resolveHostName(userId: string): Promise<string> {
  try {
    const svc = createServiceRoleClient()
    const { data } = await svc.auth.admin.getUserById(userId)
    const user = data?.user
    if (!user) return 'your host'
    const meta = (user.user_metadata ?? {}) as Record<string, unknown>
    const fullName =
      (typeof meta.full_name === 'string' && meta.full_name) ||
      (typeof meta.name === 'string' && meta.name) ||
      null
    if (fullName) return fullName
    return user.email ?? 'your host'
  } catch {
    return 'your host'
  }
}
```
