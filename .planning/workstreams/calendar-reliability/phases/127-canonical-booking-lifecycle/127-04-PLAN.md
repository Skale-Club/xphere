<objective>
Fix `src/lib/mcp/tools/bookings.ts`'s `bookings_cancel` MCP tool — RESEARCH's clearest LIFE-03 gap: this handler updates `status: 'cancelled'` and returns `{ cancelled: true }` without ever calling `emitCalendarEvent`. Any org relying on a "booking cancelled" workflow (SMS notice, opportunity update, etc.) gets nothing when the cancellation happens through an MCP client (Claude, ChatGPT, or any MCP-connected agent) instead of the public cancel page or dashboard. This plan wires it to Plan 127-01's canonical `cancelBooking(ctx, bookingId, orgId)`, while preserving the existing "append reason to notes" side-write (which happens as a small separate `.update({ notes })` BEFORE the transition, so a notes-write failure never silently blocks or duplicates the actual cancellation).

Also extends the MCP `bookings_list`/`bookings_create` local `BookingStatus` zod enum (currently `z.enum(['confirmed', 'cancelled', 'no_show'])` — missing `'showed'`) to the full vocabulary, so an MCP client can filter/see `showed` bookings like it can every other status.

Purpose: closes the "cancel via MCP fires nothing" gap (Pitfall 3) and the MCP-local status vocabulary gap (LIFE-02).
Output: `bookings_cancel` emits `meeting.cancelled` on success; `bookings_list`'s status filter accepts all four real statuses.
</objective>

       ```typescript
       handler: async ({ booking_id, reason }, { auth }) => {
         const supabase = db()

         if (reason) {
           const { data: current } = await supabase
             .from('bookings')
             .select('notes')
             .eq('id', booking_id)
             .eq('org_id', auth.orgId)
             .maybeSingle()
           if (!current) return { error: 'not_found', status: 404 }
           const prev = (current.notes as string | null) ?? ''
           const notes = prev
             ? `${prev}\n\n[Cancelled via MCP] ${reason}`
             : `[Cancelled via MCP] ${reason}`
           await supabase.from('bookings').update({ notes }).eq('id', booking_id).eq('org_id', auth.orgId)
         }

         const result = await cancelBooking({ supabase, depth: 0 }, booking_id, auth.orgId)
         if (!result.ok) {
           const statusByError: Record<string, number> = {
             booking_not_found: 404,
             illegal_transition: 409,
           }
           return { error: result.error ?? 'cancel_failed', status: statusByError[result.error ?? ''] ?? 500 }
         }

         return { cancelled: true }
       },
       ```
       (This intentionally does NOT re-fetch/re-check org ownership before calling `cancelBooking` beyond the notes-update's own `.eq('org_id', auth.orgId)` guard when `reason` is present — `transition.ts::cancelBooking`'s own RPC-backed org check is now the authoritative boundary for the no-`reason` path too.)
