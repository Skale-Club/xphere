<objective>
Wire the two native `bookings.ts` cancellation paths to Plan 127-01's canonical `cancelBooking(ctx, bookingId, orgId)` from `src/lib/calendar/transition.ts`, closing two concrete gaps RESEARCH identified:

1. **Dashboard `cancelBooking(id)`** (lines ~176-211) has no current-status guard and no rows-affected check — a double-click or a race with another cancel path always fires a second `meeting.cancelled` event even though the booking was already cancelled. The canonical service's idempotent guard closes this for free.
2. **`cancelBookingByToken`** is already safe today (atomic `UPDATE ... WHERE status='confirmed' ... RETURNING`, only emits when a row comes back) but bypasses `bookings.status` directly rather than going through the one canonical path D-01 requires every writer to use. This plan preserves its token-verification security property (token match is checked BEFORE any mutation, in a read-only SELECT) while delegating the actual guarded status write + event emission to the canonical service.

`createBooking`/`createBookingInternal` are INSERT-time writers, not status transitions (there is no "previous status" to guard against) — RESEARCH does not flag their event emission as buggy (both already check the insert's own error before firing `meeting.scheduled`). This plan makes their `status: 'confirmed'` literal source from the shared `BOOKING_STATUSES` vocabulary (Plan 127-01) instead of a bare string, for LIFE-02 consistency, without altering their emission logic.

Purpose: closes the double-fire bug (Pitfall 6) and brings the two native cancellation paths onto the single canonical transition path (LIFE-01/LIFE-03).
Output: `src/app/(dashboard)/calendar/_actions/bookings.ts`'s `cancelBooking` and `cancelBookingByToken` both delegate to `transition.ts::cancelBooking`; `createBooking`/`createBookingInternal` import the shared status constant.
</objective>

       ```typescript
       export async function cancelBookingByToken(
         bookingId: string,
         cancelToken: string,
       ): Promise<ActionResult<void>> {
         const supabase = createServiceRoleClient()

         // Token verification happens here, read-only, BEFORE any mutation attempt
         // -- an invalid token or already-cancelled booking never reaches the
         // canonical transition service at all.
         const { data, error } = await supabase
           .from('bookings')
           .select('id, org_id')
           .eq('id', bookingId)
           .eq('cancel_token', cancelToken)
           .eq('status', 'confirmed')
           .single()

         if (error || !data) return { ok: false, error: 'not_found_or_already_cancelled' }

         const result = await transitionCancelBooking({ supabase, depth: 0 }, bookingId, data.org_id)
         if (!result.ok) return { ok: false, error: 'not_found_or_already_cancelled' }

         void sendCancellationEmailForBooking(bookingId).catch(() => {})

         return { ok: true, data: undefined }
       }
       ```
