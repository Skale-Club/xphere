<objective>
Fix two concrete, verified bugs in `src/app/api/xkedule/webhook/route.ts` (D-04 requires Xkedule's mirror semantics stay otherwise unchanged — this plan touches only `mapStatus`, `calendarEventFor`, and the existing-row update branch's error handling, nothing else):

1. **Pitfall 4 — event fires after a failed write.** The existing-row update branch (`await supabase.from('bookings').update(mutable).eq('id', existing.id)`) discards the `{data, error}` result entirely and falls through unconditionally to `emitCalendarEvent`. If the update fails, the event still fires describing a state change that never happened — a direct, literal violation of D-03 ("no events after failed writes").
2. **Pitfall 5 — Xkedule's own `completed` status is silently lost.** `mapStatus()` only distinguishes `cancelled` and `no_show` explicitly; Xkedule's `completed` (and every other unrecognized status) maps to native `confirmed` forever. A booking Xkedule reports as fully serviced never reaches the DB's `showed` value, so `showed`-triggered workflows (and, after Plan 127-01, the `meeting.completed` event) can never fire for Xkedule-sourced bookings.

This also fixes `calendarEventFor`'s fallback: today any unrecognized event/status combination — including the new `status === 'showed'` case this plan introduces — falls back to `meeting.rescheduled`, which is not accurate for a completion. This plan adds an explicit `status === 'showed' → meeting.completed'` branch.

Purpose: closes the last two LIFE-03 writer-category gaps (Xkedule inbound) and the last LIFE-02 vocabulary gap (Xkedule's own completion signal was being thrown away).
Output: `mapStatus`/`calendarEventFor` updated; the existing-row update branch checks its own result before emitting; `tests/xkedule-webhook.test.ts` (new) covers both fixes plus a regression pass over the insert branch (already correct today).
</objective>
