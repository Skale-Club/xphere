# Plan 59-01: Server actions + lib refactor

## Tasks (all in one commit — tight coupling)

### Task 1: Create `src/app/(dashboard)/integrations/twilio/numbers-actions.ts`

CRUD server actions with Zod validation per CONTEXT.md decisions. Exports:

- `listTwilioNumbers()` — lists active numbers for current org, ordered by `is_default DESC, friendly_name ASC`. Returns `[]` if no integration.
- `createTwilioNumber(input)` — Zod validates; if `is_default=true`, clears prior defaults first (within the same RLS-scoped client call sequence).
- `updateTwilioNumber(id, input)` — partial-shape Zod; same default-toggle logic.
- `softDeleteTwilioNumber(id)` — sets `is_active=false`; if the deleted row was the default, no automatic promotion (operator picks the new default).
- `setDefaultTwilioNumber(id)` — single-purpose action: clears prior default + sets this row as default. Surfaces clear error if the row is inactive.

Also exports `TwilioPhoneNumberRow` type alias from `database.ts`.

Revalidates `/integrations`, `/integrations/twilio`, `/settings/calls` on every mutation.

### Task 2: Add helper `getDefaultTwilioNumberForOrg(orgId)` in `actions.ts`

Service-role lookup helper consumed by `voice.ts` and `send-sms.ts`. Returns `TwilioPhoneNumberRow | null`. Placed near the bottom of `actions.ts` next to other internal helpers.

### Task 3: Refactor `src/lib/twilio/voice.ts`

- `resolveTwilioCredentialsForOrg(orgId, options?: { phoneNumberId?: string })`:
  - If `options.phoneNumberId` is provided, look up that number (service-role, scoped by org)
  - Otherwise, look up the default via `getDefaultTwilioNumberForOrg(orgId)`
  - If neither yields a number, fall back to `config.from_number` (legacy compat)
  - Return `TwilioVoiceCredentials | null`

- `resolveTwilioOrgByToNumber(toNumber)`:
  - First query: join `twilio_phone_numbers tpn` ON `tpn.organization_id = integrations.organization_id` WHERE `tpn.e164 = ? AND tpn.is_active = true AND integrations.provider = 'twilio' AND integrations.is_active = true`
  - If no row found, fallback to the existing `WHERE config->>'from_number' = ?` query
  - Both branches produce identical `{ orgId, creds }` shape

### Task 4: Refactor `src/lib/twilio/send-sms.ts`

- `resolveTwilioCredentials(ctx, options?: { fromNumberId?: string })`:
  - If `fromNumberId` is provided, look up that number in `twilio_phone_numbers` (must be `is_active=true` and `capability_sms=true`)
  - Otherwise, look up the org's default; require `capability_sms=true`
  - Fallback to `config.from_number` only if no row found at all
  - Clear error if SMS capability is missing on the resolved number: `"Phone number {e164} does not have SMS capability enabled."`

- `sendSms(params, ctx)` — accept optional `params.fromNumberId` (string) and forward to `resolveTwilioCredentials`. Type signature unchanged (still `Record<string, unknown>`).

### Task 5: Update `src/app/(dashboard)/integrations/twilio/actions.ts`

- `TwilioIntegrationView` adds `numbers: TwilioPhoneNumberRow[]`
- `getTwilioIntegration` fetches numbers in parallel with the integration row; populates `view.numbers`
- `smsConfigured = hasAccountSid && hasAuthToken && numbers.some(n => n.is_active && n.capability_sms)` — drops the `fromNumber` boolean check
- `voiceConfigured = hasAccountSid && hasApiKeySid && hasApiKeySecret && Boolean(twimlAppSid) && numbers.some(n => n.is_active && n.capability_voice)` — adds voice capability gate
- `saveTwilioIntegration({fromNumber?})`: `fromNumber` is no longer written to `config.from_number` from this action; the input field stays in the type for backwards-compat callers but is ignored. UI calls `numbers-actions.ts` going forward.
- `testSendSms({to, fromNumberId?})`: resolves the `From` via the new path; defaults to org's default number if not provided
