Opps Widget — Smoke Test

Manual browser verification for Phase 4. The chat widget should appear in the bottom-right corner.

Note: This page must be served from the same origin as the API (e.g., http://localhost:4267). Open via npm run dev then visit http://localhost:4267/widget-test.html. Replace data-token below with a real widget_token from your database.

Verification Checklist

  1. Floating bubble appears in bottom-right corner (56px circle, dark background)
  2. On first load (no localStorage): bubble pulses twice after 1.2s delay
  3. Clicking bubble: chat panel opens with 200ms animation, bubble icon becomes X
  4. Panel header shows "AI Assistant" with avatar initial "A"
  5. Empty state shows "Hi! How can I help?" centered in message area
  6. Input field focused automatically when panel opens
  7. Type a message and press Enter or click send button: user bubble appears, typing dots appear
  8. Typing dots animate (3-dot bounce) while waiting for response
  9. Full AI response appears as assistant bubble when stream completes
  10. Input re-enables and clears after response received
  11. Clicking X bubble: panel closes with 160ms animation, bubble icon returns to chat
  12. Reload page: session persists (same sessionId reused, no new session event on first message)
  13. Greeting: after the configured delay, a "Write a message…" pill slides in beside the bubble (with a greeting bubble above it)
  14. Typing in the greeting pill + Enter/send opens the panel and sends that message
  15. Clicking the greeting × dismisses it; it stays gone on reload (sessionStorage)
  16. JS API: in the console, Opps.open(), Opps.close(), and Opps.sendMessage('hi') all work

Context Forwarding Checklist (Phase 133, CTX-03)

  1. The script tag below has data-context-endpoint set — on first send, the widget fetches a token from that path SAME-ORIGIN (credentials: 'same-origin'), never prefixed with the widget's apiBase
  2. In DevTools Network tab: confirm a request to the context endpoint happens before/around the chat POST
  3. The POST to /api/chat/<token> body includes commerce_context only when a token was successfully fetched/cached — if the endpoint 404s or is unreachable, the body has NO commerce_context and the chat still sends (fail-soft, never blocks)
  4. In the console: Opps.setContext('<payload>.<sig>') replaces the cached token — the next chat POST forwards the new value
  5. A commerce SSE event with action: 'cart_created' clears the cache, forcing a re-fetch from data-context-endpoint on the next send

Commerce Re-dispatch Checklist (Phase 134, CRT-04)

  1. Every commerce SSE event the widget receives is re-dispatched to the host page as window.CustomEvent('xphere:commerce'), with detail: { action, cartId, itemCount, sig } matching contract §6 exactly — this is what the stuscle storefront bridge listens for (cart_updated → refreshAgentCart, cart_created → adoptAgentCart)
  2. The Phase 133 cache-clear behavior on cart_created still fires (additive, not replaced) — the widget forces a context re-fetch on the next send
  3. Paste this in the console before sending a cart-affecting message, to observe the re-dispatch live: 
    window.addEventListener('xphere:commerce', e => console.log('xphere:commerce', e.detail))
  4. Trigger a cart-write tool call (e.g. "add this to my cart") that results in a cart_created frame, then another that results in cart_updated — confirm both log with the correct action, cartId, itemCount, and sig keys
  5. Deferred: this checklist item requires a live stuscle storefront + agent stack wired end-to-end. Like the 133-03 context-forwarding checklist, verify manually once both stacks are running together; not required for the automated gate.
